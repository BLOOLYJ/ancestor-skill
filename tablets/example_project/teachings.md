# 祖训 —— 赛博香火铺

> 前人之言，后人之鉴。以下规范乃血泪凝结，违者自负。

## 技术规范

- 所有金额字段使用 `Decimal` 类型，禁止使用 `float`（上次用 float 算香火钱，少了一分钱，被用户投诉到消协）
- API 响应统一使用 `{"code": 0, "message": "ok", "data": {...}}` 格式
- 数据库查询必须有超时设置，默认 5 秒
- Redis 缓存 key 必须带前缀 `cyber_incense:`，TTL 不超过 24 小时

## 代码风格

- 函数命名使用 `snake_case`
- 类命名使用 `PascalCase`
- 常量全部大写，用下划线分隔
- 每个 API 端点必须有 docstring，包含参数说明和返回值说明
- 不允许在代码中硬编码配置值，全部走环境变量

## 红线禁区

- **绝对不要直接操作 `offerings` 表的 `status` 字段**，必须通过 `OfferingStateMachine` 类
- **不要在非事务上下文中修改用户余额**，否则扣钱和加香火可能不一致
- **不要删除 `legacy_blessing_handler.py`**，虽然代码很丑但它处理了 17 种历史遗留的祈福格式，没有文档，只有它自己知道怎么解析
- **不要升级 `cryptography` 库到 42.x**，和当前的 Python 3.11 有兼容问题

## 推荐做法

- 新增 API 时先写 OpenAPI schema，再写实现
- 复杂业务逻辑先画状态机图，确认流转无误再编码
- 数据库迁移必须写 rollback 脚本
- 上线前跑一遍 `pytest --cov` 确保覆盖率不低于 80%
