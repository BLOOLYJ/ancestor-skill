# 三叩九拜 🙇

> 仪式第四步：九项系统化质量检查——整个仪式的核心

## 目标

三叩 = 三大维度（正确性、健壮性、可维护性）
九拜 = 九项具体检查

这不是形式主义，是系统化的质量保证。每一拜都要认真检查、诚实评估。

## 评估标准

- ✅ 通过：该项无问题，已经考虑到位
- ⚠️ 需注意：存在潜在风险，但不影响核心功能，需要留意
- ❌ 有问题：存在明确问题，建议在执行前解决

## 九项检查

### 第一叩：正确性 —— 你做的是对的吗？

**一拜：需求理解**
- 重新复述一遍需求，确认理解无误
- 是否有歧义的地方？
- 是否遗漏了隐含需求？
- 判断：用户说的和你理解的是同一件事吗？

**二拜：逻辑自洽**
- 方案中有没有逻辑矛盾？
- 数据流是否通畅？
- 状态转换是否完整（有没有死胡同或无限循环）？
- 判断：方案在逻辑上是否站得住脚？

**三拜：边界条件**
- 空值 / null / undefined 处理了吗？
- 极值（最大、最小、零）考虑了吗？
- 并发场景考虑了吗？
- 异常输入会怎样？
- 判断：正常路径之外的路径是否安全？

### 第二叩：健壮性 —— 你做的能扛得住吗？

**四拜：错误处理**
- 异常是否被正确捕获？
- 有没有降级方案？
- 失败时是否有合理的反馈？
- 是否需要重试机制？
- 判断：出了错会怎样？用户体验会崩吗？

**五拜：安全合规**
- 是否有 SQL 注入 / XSS / CSRF 风险？
- 权限检查是否到位？
- 敏感数据是否加密或脱敏？
- 是否有数据泄露的可能？
- 判断：会不会被坏人利用？

**六拜：性能表现**
- 时间复杂度是否合理？
- 是否有 N+1 查询？
- 是否有不必要的重复计算？
- 内存占用是否可控？
- 判断：数据量翻 10 倍还能跑吗？

### 第三叩：可维护性 —— 后人能接得住吗？

**七拜：代码质量**
- 命名是否清晰表意？
- 函数是否遵循单一职责？
- 是否有重复代码？
- 结构是否合理？
- 判断：半年后你自己还能看懂吗？

**八拜：兼容迁移**
- 是否向后兼容？
- API 变更是否需要版本号？
- 数据库变更是否需要迁移脚本？
- 是否影响其他服务/客户端？
- 判断：上线后老版本还能正常工作吗？

**九拜：可观测性**
- 关键操作是否有日志？
- 是否有指标可以监控？
- 出了问题能多快定位？
- 是否可以快速回滚？
- 判断：凌晨 3 点出了 bug，你能 10 分钟内定位吗？

## 输出格式

```
🙇 三 叩 九 拜 ，诸 项 检 视 🙇

第一叩「正确性」
  一拜 需求理解：{✅/⚠️/❌} {具体说明}
  二拜 逻辑自洽：{✅/⚠️/❌} {具体说明}
  三拜 边界条件：{✅/⚠️/❌} {具体说明}

第二叩「健壮性」
  四拜 错误处理：{✅/⚠️/❌} {具体说明}
  五拜 安全合规：{✅/⚠️/❌} {具体说明}
  六拜 性能表现：{✅/⚠️/❌} {具体说明}

第三叩「可维护性」
  七拜 代码质量：{✅/⚠️/❌} {具体说明}
  八拜 兼容迁移：{✅/⚠️/❌} {具体说明}
  九拜 可观测性：{✅/⚠️/❌} {具体说明}

叩拜完毕，请示卦象……
```

## 要点

- 每一项都必须给出判断，不能跳过
- 说明要具体，不能只说"没问题"，要说清楚为什么没问题或者有什么问题
- 诚实评估，不要为了好看给 ✅
- 对于标注 ⚠️ 和 ❌ 的项，要在后续步骤中给出具体建议
- 这九项检查的框架可以适应任何类型的代码变更
